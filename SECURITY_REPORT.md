## 0. ملاحظة منهجية مهمة (واقع المشروع مقابل الطلب)

طُلب تنفيذ: install/build، تشغيل Unit/Integration/E2E، اختبار أداء وتوافق عبر متصفحات/أجهزة. الواقع التقني للمشروع:

- **لا يوجد build system ولا `package.json` ولا أي إطار اختبار** — المشروع HTML/CSS/JS ثابت يُنشر مباشرة على GitHub Pages، و`CONTEXT.md` ينص صراحة على **عدم** إدخال npm/node/webpack.
- لذلك "install/build" و"تشغيل الاختبارات الحالية" **غير منطبقة** (لا يوجد ما يُبنى أو اختبارات قائمة).
- اختبار الأداء و**التوافق عبر المتصفحات/الأجهزة الحقيقية** يتطلب متصفحات فعلية / أجهزة / أدوات مثل Lighthouse أو BrowserStack وهي **غير متاحة داخل بيئة التنفيذ المعزولة** هذه. النتائج المتعلقة بها أدناه مبنية على **مراجعة الكود الثابتة** لا على قياس حي.

ما تم إنجازه فعلياً بدلاً من ذلك:
- تحليل أمني عميق يدوي لكامل `app.js` وملفات SQL وملف Edge Function.
- **إضافة أول مجموعة اختبارات للمشروع** (بدون أي تبعيات npm، باستخدام مُشغّل Node المدمج) تختبر الكود المُنتَج فعلاً.
- إصلاح الثغرات القابلة للإصلاح بأمان دون كسر التطبيق.

## 2. المشاكل مرتبة حسب الأولوية

### 🔴 CRITICAL

**C1 — لا يوجد migration ولا سياسات RLS لجدول `messages`** *(أُصلح)*
الكود يقرأ/يكتب/يحذف من `public.messages` لكن لا يوجد أي ملف migration له. إن لم تُضبط RLS يدوياً على Supabase، فإن المفتاح العام (publishable/anon) يسمح **لأي شخص بقراءة كل رسائل كل المستخدمين** (تسرب بيانات شامل) و**حذف أو تعديل أي رسالة** (`messagesAPI.delete/reply/markRead` تستخدم `.eq('id', …)` بلا تحقق ملكية).
**الإصلاح:** أُضيف `supabase/migrations/20260529000000_create_messages.sql` بسياسات صارمة:
- المستلم فقط يقرأ صندوقه؛ والعامة يقرؤون **فقط** الرسائل التي لها رد عام.
- المُرسل لا يستطيع انتحال `sender_id` لغيره (`sender_id is null or sender_id = auth.uid()`).
- التعديل/الحذف للمستلم فقط.
- قيود `CHECK` على طول المحتوى (1..1000) خادمياً.

**C2 — تخزين كلمة المرور كنص صريح في `localStorage`** *(أُصلح)*
عند التسجيل في وضع Supabase كان الكود يدفع `{username, email, password}` إلى `state._localUsers` ويحفظها في `localStorage`. أي وصول (XSS، جهاز مشترك، إضافة متصفح) يكشف كلمة المرور.
**الإصلاح:** أُزيل تخزين كلمة المرور نهائياً في وضع Supabase (يبقى فقط في وضع offline-only الذي لا يملك خادماً للتحقق).

**C3 — migration جدول `blocks` معطوب** *(أُصلح)*
كان يشير إلى جدول غير موجود `public.users` ويستخدم `uuid_generate_v4()` (امتداد قد يكون غير مفعّل) → الـ migration يفشل ⇒ ميزة الحظر قد لا تعمل على قاعدة جديدة.
**الإصلاح:** التأشير على `public.profiles`، استخدام `gen_random_uuid()` المدمج، وجعل السياسات idempotent.

### 🟠 HIGH

**H1 — تسرب هوية المُرسل يكسر "المجهولية"** *(موثّق + توصية)*
`messagesAPI.getInbox` يجلب `sender_id`، فيستطيع المستلم (عبر استجابة الـ API) معرفة هوية أي مُرسل **مسجّل دخوله** رغم أن الواجهة لا تعرضها. هذا يناقض جوهر التطبيق. السبب أن ميزة الحظر مبنية على معرفة `sender_id`.
**التوصية (لم تُنفّذ لتفادي كسر السلوك):** نقل فلترة المحظورين إلى دالة `SECURITY DEFINER` في Postgres لا تُرجع `sender_id` للعميل، أو تخزين بديل أحادي الاتجاه (hash مملّح) للمُرسل يُستخدم للحظر دون كشف الهوية.

**H2 — `escHtml` لم يكن يهرّب علامة الاقتباس المفردة** *(أُصلح)*
اسم المستخدم يُحقن داخل سمات `onclick="...('${value}')"` ذات اقتباس مفرد، و`escHtml` القديم لم يهرّب `'` ولا `` ` ``. اسم المستخدم المُعيَّن عبر REST API يتجاوز قيد `[A-Za-z0-9_]` العميلي ⇒ احتمال XSS/attribute breakout.
**الإصلاح:** توسيع `escHtml` ليشمل `'` و`` ` ``، وتطبيقه على `shareUrl`/avatar `src`/`alt`/أسماء المستخدمين في كل سياقات السمات والـ toasts (التي تستخدم `innerHTML`).

**H3 — غياب Rate Limiting / حماية Spam خادمية** *(تخفيف جزئي)*
لا يوجد أي تحديد لمعدل الإرسال؛ يمكن إغراق أي مستخدم بالرسائل، ولا CAPTCHA.
**الإصلاح الجزئي:** أُضيف throttle عميلي بسيط (1.5 ثانية بين الإرسالات) كحدّ أدنى.
**التوصية:** Rate limiting حقيقي عبر Postgres trigger / Edge Function (مثلاً حدّ لكل مستلم/نافذة زمنية)، وإضافة CAPTCHA أو proof-of-work للمرسلين غير المسجّلين.

### 🟡 MEDIUM

- **M1 — تحقق الطول عميلي فقط:** `maxlength="300"` يمكن تجاوزه بنداء REST مباشر. *(أُصلح: قيد `CHECK` خادمي ≤1000 + `validateMessage` عميلي).*
- **M2 — فحص المالك (`abadihdar@gmail.com`) من جهة العميل:** مقبول لإخفاء زر الواجهة فقط؛ لوحة التحليلات تعرض بيانات محلية فقط فلا تسرّب. التوصية: أي بيانات تحليلية حقيقية يجب أن تُحمى بـ RLS/دالة خادمية لا بالشرط العميلي.
- **M3 — Edge Function بأعمدة غير موجودة** *(أُصلح):* كان يستعلم `username`/`bio`/`recipient_id` بينما المخطط `full_name`/`receiver_id`. صُحّحت الأعمدة.
- **M4 — رفع الصورة يُتحقق منه عميلياً فقط** (النوع/الحجم): التوصية فرض سياسات Storage وقيود نوع/حجم على Supabase.

### 🟢 LOW

- **L1** — عدّاد "غير المقروء" في الشريط العلوي يحسب من `state._localMessages` الفارغ في وضع Supabase ⇒ الشارة تبقى 0 (خلل وظيفي بسيط).
- **L2** — ملفات غير مستخدمة: `replace.js` (سكربت توليد قديم يشير لمسار macOS محلي)، `landing.html`، `auth-twitter.html` غير مرتبطة بـ `index.html`. يُنصح بحذفها لتقليل الالتباس.
- **L3** — CORS في Edge Function = `*` (مقبول لنقطة عامة، لكن يُفضّل تقييده).
- **L4** — `user.bio` يُقرأ في الواجهة لكن لا عمود `bio` في `profiles` ⇒ يظهر النص الافتراضي دائماً.

## 3. تحليل حسب فئة الطلب

- **XSS:** كان السطح الرئيسي هو الحقن في السمات (H2) + `innerHTML` في `showToast`/الواجهات. الآن كل مدخلات المستخدم تمرّ عبر `escHtml` المُقوّى. محتوى الرسائل والردود كان يُهرَّب أصلاً. ✅
- **SQL Injection:** غير قابل للاستغلال — كل الاستعلامات عبر Supabase query builder (مُعاملات مُهيّأة)، ولا SQL خام من مدخلات المستخدم. ✅
- **CSRF:** غير منطبق فعلياً — Supabase يستخدم Bearer tokens في الـ headers لا cookies. ✅
- **Rate Limiting / Spam:** نقطة ضعف حقيقية (H3) — تخفيف عميلي فقط، يتطلب حلاً خادمياً.
- **Privacy / Anonymity:** الخطر الأبرز (H1) — `sender_id` يصل للمستلم. موثّق مع مسار إصلاح.
- **Data leakage:** كلمة المرور في localStorage (C2 ✅)، وغياب RLS لـ messages (C1 ✅).
- **Auth/Authorization bypass:** الاعتماد كان على RLS غير الموجودة (C1). بعد إضافة السياسات، يصبح التحكم خادمياً سليماً. فحص المالك العميلي (M2) ليس حدّ أمان حقيقي.
- **الأداء/التوافق/الأجهزة:** لم يُقَس حياً (لا أدوات في البيئة). ملاحظات من قراءة الكود: `initParticles` (25 عنصر) و`initCursor` بحلقة `requestAnimationFrame` دائمة قد تستنزف البطارية على الموبايل؛ لا تنظيف للمستمعين عند إعادة الرسم (إعادة بناء `innerHTML` كاملة في كل تنقل) — مقبول للحجم الحالي لكنه نمط غير مثالي. لا تسريبات ذاكرة صريحة عُثر عليها.
